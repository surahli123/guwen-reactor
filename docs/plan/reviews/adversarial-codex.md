**Findings**

- **P0 — Unsupported details can false-pass the hard gate.** In Task 4.2, `UNSUPPORTED_DETAIL` is counted as factual but not critical; `CRITICAL_LABELS` only includes motivation/contradiction. Yet D2 expects unsupported detail to block via `unsupported_critical_claims`. Result: “Guan Ning loudly argues...” may be labeled unsupported detail and still export. [implementation-plan.md](/Users/surahli/Documents/guwen-reactor/docs/plan/implementation-plan.md:172), [implementation-plan.md](/Users/surahli/Documents/guwen-reactor/docs/plan/implementation-plan.md:573)

- **P0 — Prompt-injection drift is not actually wired into the export gate.** D6 expects `prompt_injection_attempts` to block, but `evaluate_claims` does not count it and `export_requires_met` has no zero/bool gate for it. The test can pass only if `evaluate_run` adds unstated behavior. [implementation-plan.md](/Users/surahli/Documents/guwen-reactor/docs/plan/implementation-plan.md:438), [implementation-plan.md](/Users/surahli/Documents/guwen-reactor/docs/plan/implementation-plan.md:614)

- **P0 — “All cited fact IDs valid” is a hard-gate requirement with no implementation task.** The global hard gate requires valid cited fact IDs, and v3 fallback depends on it, but Phase 4 never defines `valid_fact_id_check_passed`. Citation spoof with nonexistent IDs is therefore not covered. [implementation-plan.md](/Users/surahli/Documents/guwen-reactor/docs/plan/implementation-plan.md:41), [Guwen_Reactor_Build_Ready_Spec_v3.md](/Users/surahli/Documents/guwen-reactor/docs/planning/guwen-v3/Guwen_Reactor_Build_Ready_Spec_v3.md:707)

- **P0 — Coverage has no trustworthy source of supported fact IDs.** `beat_coverage(supported_fact_ids, scene)` assumes fact-level support exists, but the evaluator returns labels, not entailed fact IDs. If coverage uses self-reported IDs, citation spoof breaks it; if it uses keyword support, omission detection is theatrical. [implementation-plan.md](/Users/surahli/Documents/guwen-reactor/docs/plan/implementation-plan.md:596)

- **P1 — Exact-string forbidden matching is too brittle for the claimed contradiction gate.** “Hua Xin keeps the gold” blocks; “Hua Xin pockets the gold,” “keeps the gold for himself,” or “does not throw it away” likely miss the forbidden set. With unsupported-detail not critical, paraphrased contradictions can false-pass. [implementation-plan.md](/Users/surahli/Documents/guwen-reactor/docs/plan/implementation-plan.md:560)

- **P1 — The baseline can become theater.** The plan targets a 30–40% baseline fail band and says “disclose + adjust” outside it, but does not define a pre-registered adjustment rule. That invites tuning prompts/scenes until the baseline looks conveniently weak. [implementation-plan.md](/Users/surahli/Documents/guwen-reactor/docs/plan/implementation-plan.md:694)

- **P1 — Metrics are denominator-gameable.** Precision over generated claims rewards terse outputs and depends on claim extraction choices. A system can improve `supported/total` by saying less. Required-beat coverage helps, but with 3 beats it is a coarse checklist, not a faithfulness metric. [Guwen_Reactor_Build_Ready_Spec_v3.md](/Users/surahli/Documents/guwen-reactor/docs/planning/guwen-v3/Guwen_Reactor_Build_Ready_Spec_v3.md:455), [implementation-plan.md](/Users/surahli/Documents/guwen-reactor/docs/plan/implementation-plan.md:44)

- **P1 — Phase 4 is schedule fiction.** Non-cuttable D4 includes claim extraction, evaluator, coverage, six-drift injector, safety, regen loop, approval/export, CLI e2e, trace/workflow integrity. That is not one day unless the code is already mostly written. [implementation-plan.md](/Users/surahli/Documents/guwen-reactor/docs/plan/implementation-plan.md:476)

- **P1 — Phase 5 measured results depend on assets Phase 4 will not realistically have.** Baseline over ~30–40 claims across 3 scenes requires two extra scenes, gold, clean outputs, baseline outputs, and scoring. Those are still TBD/open decisions. [implementation-plan.md](/Users/surahli/Documents/guwen-reactor/docs/plan/implementation-plan.md:692), [implementation-plan.md](/Users/surahli/Documents/guwen-reactor/docs/plan/implementation-plan.md:772)

- **P1 — Gold construction is the hidden epistemic weak point.** The product forbids using English translations, but the gold facts are English semantic claims. The plan does not specify how those English facts are produced without contaminating from translations, model priors, or summaries. One independent beat check is not enough to make the gold credible. [implementation-plan.md](/Users/surahli/Documents/guwen-reactor/docs/plan/implementation-plan.md:246), [Guwen_Reactor_Build_Ready_Spec_v3.md](/Users/surahli/Documents/guwen-reactor/docs/planning/guwen-v3/Guwen_Reactor_Build_Ready_Spec_v3.md:594)

- **P2 — Simpler design:** drop free-form claim extraction as the centerpiece. Make generation fill constrained fields tied to required beats: `beat_id`, `claim_text`, `source_fact_ids`, `assertion_type`, `hedging`. Deterministically validate coverage, forbidden deltas, missing/invalid fact IDs, unhedged motive/emotion fields, safety, manifest, and human approval. Use LLM/human review for semantic entailment, explicitly advisory. That is less grand but much more defensible.

- **P2 — Strategic risk:** MCP/skills/ADK wrappers dilute the capstone. The winning demo is educator artifact + trust gate + human approval. A shaky “deterministic faithfulness eval” is worse than an honest constrained-source audit harness.

**Single biggest risk:** the plan sells a rigorous measured faithfulness evaluator, but the actual P0 mechanism can both false-pass serious unsupported claims and false-block valid interpretations; a skeptical product data scientist will see the eval as staged rather than measured.
